# Feature Landscape

**Domain:** AI Meeting Companion Panel (Zoom Sidebar)
**Researched:** 2026-03-20

---

## Table Stakes

Features that must exist for the product to function. Without these, the panel is useless.

| Feature | Why Expected | Complexity | Notes |
|---------|--------------|------------|-------|
| Live task feed | Core value prop — see work happening during the meeting | Medium | WebSocket stream of classified intents displayed as cards with status badges |
| Meeting context display | Must know WHO is in the meeting and WHAT project this relates to | Medium | Load from Calendar attendees + Google Contacts + Linear projects on meeting start |
| WebSocket connection to engine | Real-time bidirectional communication is the backbone | Medium | FastAPI WebSocket server, react-use-websocket client, reconnection handling |
| Intent classification | Engine must understand what's being discussed and what to do about it | High | Multi-model fallback chain (Gemini -> OpenAI -> Claude -> Keywords) via LiteLLM |
| Agent status indicators | Must show which fleet agents are working and on what | Low | Simple status badges: idle, working, completed, failed |
| Connection status indicator | User must know if panel is connected to engine | Low | Green/yellow/red dot. WebSocket readyState mapping. |
| Post-meeting summary | Recap of decisions, action items, follow-ups | Medium | Generated by LLM from accumulated transcript + classified intents |

## Differentiators

Features that make this copilot uniquely valuable vs generic meeting notes tools.

| Feature | Value Proposition | Complexity | Notes |
|---------|-------------------|------------|-------|
| Project-aware routing | Action items auto-route to the correct Linear project based on who's in the meeting | High | Cross-reference attendee email -> Google Contacts -> client profile -> Linear project |
| Real-time agent spawning | Fleet agents execute work DURING the meeting, not after | High | asyncio subprocess spawning of Claude Code agents with task context |
| Quick action buttons | One-tap triggers: Delegate Task, Create Proposal, Research This, Draft Email, Check Domain | Medium | Each button fires a WebSocket command to engine, which dispatches to the right agent/skill |
| Internal vs client detection | Adjusts behavior based on meeting type (internal team vs client call) | Medium | Attendee classification: known team members = internal, unknown emails = client |
| Multi-project switching | During internal meetings, detect "Now about the copilot..." and re-route to correct project | High | Requires continuous context tracking within transcript, not just per-meeting |
| Prior meeting context | "Last time you met with X, you discussed: [topics]. Open items: [list]" | Medium | Query last 3 Fireflies transcripts for same attendees, summarize |
| Auto follow-up email | Draft and send post-meeting email with action items and decisions | Medium | Gmail API send with LLM-generated content, attendee addresses from Calendar |
| Linear project auto-creation | New client? Auto-create Linear project with initial issues from the meeting | Medium | Linear GraphQL mutation, triggered when no existing project found for attendee |

## Anti-Features

Features to explicitly NOT build. These are tempting but wrong.

| Anti-Feature | Why Avoid | What to Do Instead |
|--------------|-----------|-------------------|
| Full transcript display in panel | Panel is 300-400px wide. Full transcript is unreadable and distracting. Fireflies already shows the transcript. | Show classified intents and key decisions only. Link to Fireflies for full transcript. |
| Voice commands / speech input | Zoom already captures audio. Adding a second audio stream creates echo, confusion, privacy issues. | Use text-based quick action buttons in the panel. |
| Video/screen analysis | Massively complex, GPU-intensive, and low ROI for task extraction vs transcript analysis. | Transcript-only for v1. Screen analysis is a v3+ exploration. |
| Chat with the copilot | Building a chat UI inside a sidebar is awkward UX. Julian should focus on the meeting, not typing to an AI. | Quick action buttons for common commands. Free-form input only via a small command bar. |
| Meeting recording/playback | Fireflies already records. Duplicating this adds storage costs and legal complexity. | Link to Fireflies recording from post-meeting summary. |
| Participant video feeds | Not possible in Zoom Apps iframe. Zoom doesn't expose video streams to apps. | N/A — not technically feasible anyway. |
| Google Meet support (v1) | Doubles the integration surface. Google Meet has a completely different extension model. | Zoom-only for v1. Google Meet is v2 scope (different app architecture). |
| Complex settings panel | Over-engineering. One user (Julian). Settings can be env vars or a config file. | Hardcode sensible defaults. Config file for customization. |

## MVP Recommendation

For MVP (Phase 1-2), prioritize:

1. **WebSocket connection** (panel <-> engine) — everything depends on this
2. **Live task feed** with status badges — core UI that makes the panel useful
3. **Intent classification** with existing multi-model chain — already partially built
4. **Connection status indicator** — must-have for reliability trust
5. **Meeting context display** — who is in the meeting, what project

Defer to post-MVP:
- **Multi-project switching**: Requires sophisticated NLP context tracking. Phase 3+.
- **Auto follow-up email**: Needs careful template design and send confirmation UX. Phase 3.
- **Linear project auto-creation**: Edge case (new clients only). Phase 3.
- **Prior meeting context**: Nice-to-have, not blocking. Phase 3.
- **Quick action buttons**: Build 1-2 for MVP (e.g., "Create Issue"), expand in Phase 3.
