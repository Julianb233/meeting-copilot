
    2. `async test_full_meeting_flow()` function:
       Uses FastAPI TestClient (from httpx or starlette.testclient) if available, otherwise uses direct bridge calls:

       **Phase 1: Meeting Start**
       - Create WatcherEvent with event_type=MEETING_START
       - Send to bridge.handle_event() or POST /api/watcher/event
       - Assert response status is "context_loaded"
       - Assert meeting_id is set on bridge
       - Print context load time and attendee count
       - Verify meeting_type is "client" (external attendee present)

       **Phase 2: Transcript Chunks**
       - Send sentences in 2 batches (simulating 30-second polling intervals):
         - Batch 1: sentences 0-3
         - Batch 2: sentences 4-7
       - For each batch, create WatcherEvent with event_type=TRANSCRIPT_CHUNK
       - Send to bridge.handle_event()
       - Assert response status is "processed"
       - After both batches, check manager.state:
         - state.transcript_chunks should have 8 entries
         - state.intents should have >= 1 entry (at least the "fix mobile responsive" or "create Stripe integration" should be detected)
       - Print detected intents count and details

       **Phase 3: Meeting End**
       - Create WatcherEvent with event_type=MEETING_END
       - Send to bridge.handle_event()
       - Assert response status is "ended"
       - Print summary and follow-up email (should be drafted but NOT actually sent — check for dry_run or mock the send)
       - Verify bridge.active_meeting_id is None (reset after meeting end)

       **Validation Checks:**
       - Context was loaded (meeting_context is not None during phases 1-2)
       - At least 1 intent was detected from the transcript
       - Meeting classified as "client" (sean@hafniafin.com is external)
       - Summary was generated with at least 1 action item
       - Follow-up email was drafted (not necessarily sent)
       - Bridge state is clean after meeting end
