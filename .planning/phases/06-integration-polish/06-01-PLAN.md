<objective>
Bridge the existing meeting-watcher v2 script to the copilot engine via REST API.

Purpose: The meeting-watcher v2 (/opt/agency-workspace/scripts/meeting-watcher.py) already detects meetings, polls Fireflies transcripts, and classifies sentences. Instead of duplicating this logic, the copilot engine receives events from the watcher via a REST bridge. The watcher sends meeting_start (with attendees), transcript_chunk (new sentences), and meeting_end events. The engine processes them through the full pipeline (context assembly, intent detection, routing, orchestration, follow-up email).

Output: WatcherBridge module and POST /api/watcher/event endpoint that the meeting-watcher can call.
</objective>

  <name>Task 1: Create WatcherBridge module</name>
  <files>engine/bridge/__init__.py, engine/bridge/watcher_bridge.py</files>
  <action>
    Create `engine/bridge/__init__.py` as empty package init.

    Create `engine/bridge/watcher_bridge.py` with:

    1. `WatcherEventType` enum (str, Enum): `MEETING_START = "meeting_start"`, `TRANSCRIPT_CHUNK = "transcript_chunk"`, `MEETING_END = "meeting_end"`

    2. `WatcherEvent` Pydantic model:
       - `event_type: WatcherEventType`
       - `meeting_id: str` — Fireflies meeting ID or calendar event ID
       - `meeting_title: str | None = None`
       - `attendee_emails: list[str] = Field(default_factory=list)` — from calendar event
       - `sentences: list[dict] = Field(default_factory=list)` — for transcript_chunk (each dict has speaker, text, timestamp keys)
       - `timestamp: str | None = None`

    3. `WatcherBridge` class:
       - `__init__(self)`: store reference to the ws_handler.manager (import at call time to avoid circular imports)
       - `active_meeting_id: str | None` — tracks currently active meeting
       - `meeting_context: UnifiedMeetingContext | None` — stored after assembly

       - `async handle_event(self, event: WatcherEvent) -> dict`:
         Main dispatch method. Routes by event_type:

         **meeting_start:**
         - Store meeting_id as active
         - Call `assemble_meeting_context(event.attendee_emails, event.meeting_title)`
         - Store result in self.meeting_context
         - Update manager.state with context info (set attendees, meeting title on manager.state.context)
         - Broadcast context_loaded event via manager.broadcast
         - Return `{"status": "context_loaded", "meeting_id": event.meeting_id, "attendees": len(event.attendee_emails), "load_time": ctx.load_time_seconds}`

         **transcript_chunk:**
         - Validate meeting is active (return error if not)
         - Feed sentences into manager by calling `manager._process_transcript(event.sentences, event.meeting_title or self.meeting_context.meeting_title)`
         - Return `{"status": "processed", "sentences": len(event.sentences)}`

         **meeting_end:**
         - If meeting not active, return early
         - Trigger post-meeting processing: generate summary and send follow-up email
         - Import generate_meeting_summary and draft_followup_email/send_followup_email from intelligence modules
         - Build intents list from manager.state.intents
         - Generate summary, draft email, send email
         - Clear active_meeting_id and meeting_context
         - Reset manager.state for next meeting
         - Return `{"status": "ended", "meeting_id": event.meeting_id, "email_sent": True/False}`

       All methods use standard logging. Wrap each in try/except to never crash the bridge.

    Use flat imports matching engine convention. Add `if __name__ == "__main__"` test block.
  </action>
