    ```python
    from context.models import (

    )

    from context.profiles import load_client_profile

    ```

    The assembler module should contain ZERO Pydantic class definitions. It only uses the models from context.models and the functions from the loader modules.

    **Function `assemble_meeting_context(emails: list[str], meeting_title: str | None = None, display_names: dict[str, str] | None = None) -> UnifiedMeetingContext`:**

    The key design insight from research: identity resolution must happen FIRST (need company name for Linear lookup), but Fireflies lookup uses email directly and can run in parallel with identity resolution.

    **Optimized two-phase fan-out:**

    ```
    Phase A (parallel): resolve_attendees(emails) + [fetch_meeting_history(e) for e in emails]
    Phase B (parallel, after A): for each attendee: [fetch_linear_projects(company/email) + load_client_profile(email, company) + fetch_git_activity(slug)]
    ```

    Implementation:
    1. Start timer with `time.monotonic()`.
    2. **Phase A** — run identity resolution and Fireflies history in parallel:
       ```python
       identity_task = resolve_attendees(emails, display_names)
       history_tasks = [fetch_meeting_history(e) for e in emails]
       results = await asyncio.gather(identity_task, *history_tasks, return_exceptions=True)
       identities = results[0]  # list[AttendeeIdentity] or Exception
       histories = results[1:]  # list[list[TranscriptSummary] | Exception]
       ```
    3. Handle Phase A exceptions: if identities is an Exception, create email_only fallbacks for all attendees. For each history that is an Exception, use empty list.
    4. **Phase B** — for each (email, identity) pair, run Linear + Profile + Git lookups in parallel:
       ```python
       phase_b_tasks = []
       for email, identity in zip(emails, identities):
           phase_b_tasks.append(fetch_linear_projects(
               company_name=identity.organization,
               email=email
           ))
           phase_b_tasks.append(asyncio.to_thread(
               load_client_profile,
               email=email,
               company=identity.organization
           ))
           # Git activity: need client slug from domain mapping or profile
           # Use the Linear client's domain mapping to get slug from email
           from context.linear_client import build_domain_mapping
           domain = email.split("@")[1] if "@" in email else None
           mapping = build_domain_mapping()
           slug = mapping.get(domain) or mapping.get(email)
           phase_b_tasks.append(fetch_git_activity(slug=slug) if slug else asyncio.coroutine(lambda: [])())
       phase_b_results = await asyncio.gather(*phase_b_tasks, return_exceptions=True)
       ```
       Note: `load_client_profile` is sync — wrapped with `asyncio.to_thread`. `fetch_git_activity` is async.
    5. Pair up Phase B results (every 3 results = one attendee: linear, profile, git_activity).
    6. Build AttendeeContext for each attendee from all gathered results. Use the imported AttendeeContext model directly:
       ```python
       AttendeeContext(
           identity=identity,
           meeting_history=history,
           linear_projects=projects,
           client_profile=profile,
           git_activity=git_commits,
       )
       ```
    7. For any result that is an Exception, log the error, add to `errors` list, use empty/None default.
    8. Record `load_time_seconds = time.monotonic() - start`.
    9. Log: "Context assembled for {n} attendees in {time:.1f}s ({m} errors)"
    10. Return UnifiedMeetingContext (imported from context.models, which includes the `to_classifier_prompt()` method).
